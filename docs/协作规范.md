# 项目协作规范

## 一、基本原则

1. 每完成一个功能模块，必须同步更新对应文档。
2. 每次重要代码变更后，需要在 `docs/迭代计划与TODO.md` 中更新状态。
3. `main` 分支必须保持可编译、可运行，不把已知破损版本合入 `main`。
4. 算法、UI、文档、测试按模块协作，接口先约定，避免职责混杂。
5. 每个阶段结束时提交一次阶段性 commit，并核对阶段验收标准。
6. 不加入登录、注册、数据库、后端、云服务等无关功能。
7. 先保证基本功能能在手机上运行，再进行性能优化。

## 二、分支与提交规范

- 功能开发建议从 `main` 创建短生命周期分支，例如：
  - `feat/image-picker`
  - `feat/baseline-histogram`
  - `perf/bitmap-reading`
  - `docs/test-report`
- 合并前先同步 `main`，解决冲突并完成相关测试。
- 一次提交只处理一个清晰目标，避免把算法、UI 和无关格式化混在一起。
- 推荐提交信息：
  - `feat: add image picker`
  - `feat: implement baseline histogram`
  - `docs: update project plan`
  - `test: add histogram unit test`
  - `perf: optimize bitmap pixel reading`
- 其他常用类型：`fix`、`refactor`、`build`、`chore`。

## 三、高冲突文件管理

不允许多人同时大幅修改以下文件：

- `MainActivity.kt`
- `app/build.gradle.kts`
- `gradle/libs.versions.toml`
- `app/src/main/AndroidManifest.xml`

修改前应在群内说明修改目的、负责人和预计完成时间。其他模块通过新增独立类、组件和数据模型完成，减少对入口和构建文件的争抢。

## 四、模块协作边界

- 算法负责人：维护灰度公式、频次统计、归一化、Baseline/Optimized 和算法测试；不直接修改页面布局。
- UI 负责人：通过数据类和 UI State 消费算法结果；Canvas 中不读取整图像素。
- 图片与状态负责人：维护 URI、Bitmap 解码、协程、取消和错误状态；不实现直方图公式。
- 测试负责人：维护确定性图片、正确性用例、真机性能记录和缺陷清单。
- 文档负责人：同步需求、设计、TODO、测试报告和使用说明，但技术结论必须由对应开发者复核。

## 五、代码与文档同步

每个模块完成时至少同步以下内容：

1. 在 TODO 中勾选任务并更新阶段状态；
2. 在设计文档中记录接口、关键规则或计时口径；
3. 为核心算法和边界处理添加适量中文注释；
4. 增加或更新测试；
5. 如果用户操作发生变化，更新使用说明和截图。

只有代码没有文档，或只有文档没有可运行实现，都不视为模块完成。

## 六、提交前检查

提交人应完成：

```bash
git status --short
./gradlew test
./gradlew assembleDebug
```

并确认：

- 没有提交 `build/`、`.gradle/`、`.idea/`、`local.properties`；
- 没有提交 `references/open_source/` 中的第三方仓库；
- 没有把参考仓库代码直接复制到 `app/src/main`；
- 没有调试密钥、个人路径、大型临时图片和无关二进制文件；
- 当前提交对应的 TODO 和文档已经同步。

## 七、阶段结束检查

- 对照 `docs/迭代计划与TODO.md` 检查任务、产物和验收标准。
- 在真机走通本阶段主流程。
- 记录未解决问题、风险和下一阶段负责人。
- 创建清晰的阶段性 commit；必要时创建版本标签。
- 阶段提交合入后再次确认 `main` 可编译、可运行。
