## 正文长版

这是杭州电子科技大学“软件开发实践 3”短学期的课题：**图像直方图计算及性能优化**。

最初要求很明确：在 Android 端选择一张图片，按 `gray = R×0.299 + G×0.587 + B×0.114` 统计 256 个灰度级，绘制 256×100 黑白直方图，并把核心计算压到 300ms 内。

我们 5 个人没有只停在“能交作业”：

📱 Kotlin + Jetpack Compose + Material 3 完成完整移动端体验  
⚙️ 同时实现“优先灰度化”和“统计时灰度化”，并验证结果一致  
🚀 从 Kotlin 基线一路优化到 NDK/C++：直接锁定 Bitmap、按行分块、线程私有 256-bin、最后统一归并  
📊 Xiaomi 14 上用 12.58MP 图片测试，Native v3.0、8 线程的一次封版记录为 **30.661ms**，达到 300ms 指标  
🌈 后续加入 RGB 分布、色偏、平均灰度、暗亮部占比和自然语言质量建议  
✂️ 可以框选 ROI，只分析画面局部  
📷 CameraX 实时显示亮度直方图和三色质量状态；还能定格、保存相册、调曝光/补光灯，并按建议重新拍摄  
✅ 封版功能测试表覆盖 21 项检查并记录通过

这次也尝试了 **Skill-driven Vibe Coding**。我们没有让 AI 一键“代写”，而是给 Agent 配了项目专用规则：算法公式和课程边界不能改、每段代码都要过测试门禁、TODO 和文档必须同步、Git 只能显式暂存并在验证后提交。人负责需求、判断和真机验收，Agent 加速重复工程工作。

项目里最有成就感的，不只是把 1415ms 级别的早期实现压下来，而是把性能数字变成了用户看得懂、真的能用于拍摄反馈的产品流程。

## 发布前检查

- 视频中不要出现通知内容、账号、定位、设备序列号或同学私人聊天。
- 性能数据必须带设备、图片和口径，不写“所有手机都能 30ms”。
- 标题与视频内容一致，不使用“全网第一”“吊打”等夸张表述。
- 确认人物素材获得公开发布许可；不确定时打码或替换。
- 发布后把链接补入 `demo/README.md`，再建立一个小型 Git checkpoint。
