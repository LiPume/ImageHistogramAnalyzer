# MVP 使用与验收说明

## 一、在 Android Studio 中运行

1. 使用 Android Studio 打开项目根目录。
2. 在 SDK Manager 安装 NDK `28.2.13676358`；项目已固定该版本，命令行首次构建也可自动安装。
3. 等待 Gradle Sync 完成。
4. 在 Device Manager 创建或启动 API 36（也可使用 API 23 以上）虚拟机。
5. 在顶部设备列表选择该虚拟机。
6. 点击 Run，运行 `app` 配置。
7. 首页出现“图像直方图分析”、功能说明和“选择图片”按钮即表示启动成功。

如果虚拟机相册没有图片，可直接安装项目本地 `test_pic/` 中的五张真实测试图：

```bash
ANDROID_SERIAL=emulator-5554 ./scripts/install_test_images.sh
```

课程最终验收建议生成并安装完整的 17 张图片集：

```bash
./scripts/prepare_final_test_images.sh
ANDROID_SERIAL=emulator-5554 ./scripts/install_test_images.sh test/final
```

完整测试集包含确定性黑/白/暗/亮/低对比度图、RGB/Alpha、12.58MP 性能图、1600 万像素边界、五张真实图和 WebP，清单与 SHA-256 见 `docs/测试图片清单.md`。

设备编号以 `adb devices` 为准。也可以把 JPG/PNG 直接拖到虚拟机窗口，或通过 Device Explorer 放入 `/sdcard/Pictures/`。

## 二、MVP 主流程测试

1. 在首页阅读核心能力和三步流程，点击蓝色“选择图片”按钮。
2. 在系统 Photo Picker 中选择一张图片并确认；如果设备选择器提供相机入口，也可以现场拍照后返回。静态图片流程不额外申请相册权限。
3. 选图成功后自动进入“图片分析”页并开始读取。此时只显示预览和图片信息，不会自动生成直方图。
4. 向下滚动到“选择计算方案”，任选一种：
   - “优先灰度化”：先生成整张图片的灰度数组，再统计 256 个灰度频次；
   - “统计时灰度化”：遍历彩色像素时同步计算灰度并累计频次，不保留中间灰度数组。
5. 点击“计算并绘制直方图”，等待计算完成。
6. 继续向下滚动并检查：
   - 图片预览方向和比例正常；
   - 文件名、分辨率、像素数量和 MIME 类型正确；
   - 出现横轴 0、128、255 的白底黑柱直方图；
   - 最高灰度频次大于 0；
   - “图像质量分析”显示暗部/中间调/亮部分布条、平均灰度、各区域占比、灰度标准差，以及偏暗/偏亮/低对比度/正常结论；
   - 性能卡片显示本次使用的计算方案、执行引擎和工作线程数；正常 v3.0 安装应显示“Native 多线程 v3.0”；
   - 性能卡片显示图片解码、Bitmap 批量取像素、灰度转换/融合统计、频次统计、归一化、其他开销和核心计算毫秒值；
   - Native 路径显示“多线程结果合并”耗时；只有 Native 库加载失败并回退 Kotlin 时才显示“未启用”；
   - 性能卡显示核心耗时占 300ms 预算的比例条；核心计算低于 300ms 时显示“已达到”提示。
7. 对同一张图片分别执行两个方案，确认直方图形态和最高频次一致。
8. 如需分析局部区域：
   - 在图片预览卡中点击“选择局部分析区域”；
   - 在图片上拖动生成矩形框；
   - 保持已选择的计算方案，点击“确认区域并重新分析”；
   - 检查预览卡显示“当前分析对象：局部区域”、分析尺寸、分析像素和占原图比例；
   - 查看重新生成的灰度直方图、RGB 三通道、图像质量分析和性能卡；
   - 点击“恢复全图分析”可回到原图并重新生成全图结果。
9. 点击“重新选择图片”，选择另一张图，确认旧方案和结果被清空且应用不崩溃。
   - 若在系统选择器中取消，当前页面与已有结果应保持不变；
   - 若新图片损坏、无法读取或超过 1600 万像素，应显示中文提示并保留上一张成功图片，用户可以再次选择。
10. 点击顶部“返回首页”或系统返回键，确认可以回到首页；若已有图片，首页显示“继续查看分析”，点击后应返回原分析状态。

## 三、实时相机直方图测试

1. 在首页点击“打开实时拍摄分析”。
2. 首次进入时点击“授权相机并开始预览”，按系统弹窗允许相机权限。
3. 页面应显示相机预览；如果设备没有可用摄像头或权限被拒绝，应显示中文说明并可返回首页。
4. 等待约 1 秒，检查实时结果区：
   - 出现“实时亮度直方图”；
   - 显示帧尺寸和数据源“Y 通道亮度”；
   - 显示最高亮度频次；
   - “实时质量指标”显示质量分类、平均亮度、暗部/亮部占比和标准差。
5. 点击“智能判断当前画面”，检查“智能拍摄建议”是否显示画面状态、判断依据、曝光建议、补光灯建议和拍摄建议。
6. 移动手机指向暗处、亮处或高反差场景，观察直方图和质量指标是否持续更新；需要刷新建议时再次点击“智能判断当前画面”。
7. 点击“返回首页”或系统返回键，确认可返回首页且静态图片分析功能不受影响。

说明：实时相机页为了保持预览流畅，当前使用 CameraX `YUV_420_888` 的 Y 通道作为亮度信号，每约 400ms 分析一帧；静态图片课程验收仍使用指定 RGB 灰度公式。当前版本的智能建议只给出本地规则建议，不会自动修改曝光、补光灯或保存照片。

## 四、建议测试图片

项目本地 `test_pic/` 已准备五张测试图片（仅用于本地测试，不上传公开仓库）：

- `1.png.jpg`：1080×1440，竖向高反差黑白人像；
- `2.jpg`：1620×1080，横向高噪点黑白场景；
- `3.jpg`：1440×1080，户外灰色地面与大面积阴影；
- `4.png`：496×658，日间室内外建筑与人物彩色场景；
- `5.png`：492×660，夜间商业街、强光源与大面积暗部彩色场景。

它们分别适合观察高反差、噪声、灰色纹理、普通日间色彩和夜景明暗分布对直方图形态的影响。

- 普通手机照片；
- 全黑、全白图片；
- 明显偏暗和偏亮图片；
- 横图与竖图；
- 透明 PNG；
- 小图、约 200–400 万像素图片；
- 超过 1600 万像素图片，用于确认安全上限提示。

## 五、命令行验证

```bash
./gradlew test
./gradlew assembleDebug
./gradlew lintDebug
./gradlew assembleDebugAndroidTest
```

连接一个模拟器后执行 Compose 仪器测试：

```bash
adb devices
ANDROID_SERIAL=emulator-5554 ./gradlew connectedDebugAndroidTest
```

运行显式性能基准（普通仪器测试会按设计跳过它，避免性能波动导致功能测试失败）：

```bash
ANDROID_SERIAL=emulator-5554 ./scripts/run_emulator_benchmark.sh
```

脚本输出中文 CSV；已确认的数据保存在 `docs/performance/模拟器性能测试_v3.csv`。当前 AVD 只有 1 个虚拟 CPU，线程缩放结论必须到 Xiaomi 14 再填写。

如果设备编号不是 `emulator-5554`，替换为 `adb devices` 显示的编号。

## 六、验收记录

请保存以下截图：

1. 首页；
2. 图片预览和分辨率信息；
3. 完整直方图；
4. 核心计算耗时和 300ms 判断；
5. ROI 局部区域分析结果：框选前/后预览、局部直方图、局部质量判断和恢复全图入口；
6. 实时相机页：权限页、预览页、实时直方图、实时质量指标和智能拍摄建议；
7. 如有错误，保存错误提示并记录图片类型和尺寸。

最终课程验收应在 Xiaomi 14 的固定构建上完成。按 `docs/final/07-真机测试与性能验收记录.md` 操作：17 个图片 ID 各一行，除超限图 B02 外，每行依次运行两个方案各一次，记录完整分段、最高频次、电量和 300ms 结论。历史多轮模拟器结果保留为优化稳定性旁证。

质量判断使用项目内统一阈值：暗部 `0..63`、亮部 `192..255`；平均灰度 `<85` 或暗部占比 `≥60%` 为偏暗，平均灰度 `>170` 或亮部占比 `≥60%` 为偏亮，其余情况下标准差 `<35` 为低对比度。这些结论用于课程演示和辅助判断，不等同于专业图像诊断。
