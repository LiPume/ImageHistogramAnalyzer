上述耗时来自 Debug 模拟器的一次运行，只能说明链路工作并在该次运行中低于 300ms，不能作为最终性能结论。

`PERF-04/05` 完成后再次在同一 API 36 模拟器走通真实选图链路，选择 492×660 PNG 并执行“统计时灰度化”：Bitmap 批量取像素 4.387ms、灰度转换与频次统计融合阶段 173.654ms、归一化 0.051ms、其他分配/调度 0.218ms、核心总计 178.310ms；多线程合并正确显示为“未启用”。该单次数据只用于确认分段计时从算法到 UI 的链路生效。

`PERF-11` 单线程热循环优化后，在同一模拟器、同一 492×660 PNG、同一“统计时灰度化”方案再次单次运行：批量取像素 6.826ms、融合阶段 94.298ms、归一化 0.053ms、其他开销 0.349ms、核心总计 101.525ms，最高频次仍为 10,069。相对前一次单次核心耗时约加速 1.76 倍；由于没有预热和多轮统计，这只是开发阶段方向证据，不作为正式性能结论。

## 七、v3.0 模拟器性能回归

v3.0 使用 NDK `AndroidBitmap_lockPixels` 直读 Bitmap、O3 C++ 热循环、线程私有 256-bin 和 Native 归并；Native 不可用时保留 Kotlin v2 回退。API 36/arm64 AVD 先全局预热 20 次，每个尺寸再预热 5 次并正式采样 10 次。

12,582,912 像素夹具的核心结果：

| 方案 | Kotlin v2 中位/P90/最差 | Native v3 中位/P90/最差 | 中位加速 |
|---|---:|---:|---:|
| 优先灰度化 | 95.948 / 121.883 / 245.425ms | 30.885 / 32.451 / 33.414ms | 3.11× |
| 统计时灰度化 | 64.455 / 69.353 / 74.055ms | 15.067 / 16.305 / 16.475ms | 4.28× |

AVD 的 `nproc` 和 `/proc/cpuinfo` 均显示 1 个 CPU，因此 1/2/4/6/8 线程缩放无稳定收益；这不是 Native 多线程失败，而是当前模拟器不能提供并行硬件。完整三尺寸数据、线程矩阵和真机版本占位位于 `docs/性能优化数据表.md`，机器可读明细位于 `docs/performance/模拟器性能测试_v3.csv`。

本节是可重复的 Debug 开发基准，不替代 Xiaomi 14 的固定原图、Release/接近 Release、多轮中位数验收。

## 十、已知限制

- API 23–27 使用 BitmapFactory；带 EXIF 旋转信息的图片预览方向需要在基础功能阶段继续验证。
- MVP 限制单张图片不超过 1600 万像素，以降低 Bitmap 与像素数组同时存在时的 OOM 风险。
- 模拟器已有预热、多轮、中位数/P90/最差和加速比；真机已有单次达标证据，正式多轮统计仍待最终验收前补充。
- Native 计算进行中不能被协程立即中断，但旧请求结果不会覆盖新图片。
- 当前尚未提供现场拍照入口，已列入 `BASE-09`。
