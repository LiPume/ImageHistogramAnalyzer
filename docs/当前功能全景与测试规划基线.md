## 一、项目定位

项目名称为“基于 Android 的移动端图像质量分析与直方图计算优化系统”。课程核心任务是：

1. 在 Android 端选择或拍摄图片；
2. 按公式 `gray = red × 0.299 + green × 0.587 + blue × 0.114` 计算灰度；
3. 生成 256 个灰度频次，并归一化为 0–100；
4. 绘制 256×100 逻辑尺寸的白底黑柱直方图；
5. 显示核心计算耗时，以 300ms 为目标；
6. 比较“优先灰度化”和“统计时灰度化”两种课程方案；
7. 在不改变课程算法结果的前提下进行性能优化。

当前版本在课程核心之外，增加了 RGB 分析、图像质量解释、ROI 局部分析和 CameraX 智能拍摄助手。这些增强功能用于说明直方图的实际价值，不替代课程规定的灰度算法和性能验收口径。

## 三、用户可见的完整功能

### 3.1 首页与入口

首页承担产品说明和两条主流程入口：

- “选择图片”：进入静态图片分析页并打开系统图片选择器；
- “打开实时拍摄分析”：进入 CameraX 智能拍摄助手；
- 说明 256-bin、300ms、两种算法、质量分析、ROI、RGB 和本地隐私边界；
- 所有二级页面均提供返回首页入口，系统返回键也能返回首页。

### 3.2 系统图片选择与拍照

- 使用系统 Photo Picker，不申请宽泛相册或文件存储权限；
- 支持 JPEG、PNG 和 WebP；
- Xiaomi 14 的系统选择器可直接进入系统相机拍照，拍照完成后返回同一静态分析流程；
- 用户取消选择时不报错、不清空已有内容；
- 更换图片失败时保留上一份成功图片和结果；
- 不通过文件绝对路径读取图片，只使用系统提供的 URI 和 `ContentResolver`。

说明：系统选择器是否直接显示相机入口由设备系统实现决定。没有入口时，可以先使用系统相机拍照，再从相册选择。

### 3.3 图片预览与元数据

选图后页面显示：

- 图片预览；
- 文件名；
- 分辨率；
- 像素数量；
- MIME 类型；
- 当前分析对象是全图还是 ROI 局部区域；
- ROI 模式下的局部尺寸、像素数和占原图比例。

单张图片安全上限为 1600 万像素：

- `4000×4000 = 16,000,000` 像素允许处理；
- `4001×4000 = 16,004,000` 像素在核心计算前拒绝；
- 超限、损坏、无法打开和格式异常均显示中文提示。

### 3.4 手动选择两种课程算法

选择图片后不会自动计算。用户必须选择一种算法，再点击“计算并绘制直方图”。

| 方案 | 处理流程 | 特点 |
|---|---|---|
| 优先灰度化 | 彩色像素 → 完整灰度缓冲 → 频次统计 → 归一化 → 绘制 | 流程直观，但需要额外灰度缓冲和遍历 |
| 统计时灰度化 | 读取彩色像素时同步灰度化和累计频次 → 归一化 → 绘制 | 合并转换和统计，减少中间数据与遍历 |

两种方案遵守同一灰度公式、取整规则、Alpha 规则、256-bin 和 0–100 归一化规则。逐 bin 输出不一致时，不允许报告性能提升。

### 3.5 Kotlin 与 Native 执行引擎

- Kotlin v2 保留为可运行、可测试的回退实现；
- Native v3 使用 `AndroidBitmap_lockPixels` 直接读取 Bitmap；
- C++ 按图像行分块处理；
- 每个线程使用私有 256-bin 频次数组，结束后合并，避免共享计数竞争；
- 优先灰度化在 Native 内使用 1 byte/像素灰度缓冲；
- Native 不可用或执行失败时，ViewModel 自动回退 Kotlin，不阻断主流程；
- UI 性能卡显示当前执行引擎和工作线程数。

### 3.6 灰度直方图绘制

- 横轴为灰度 0–255；
- 纵轴归一化到 0–100；
- 使用 Compose Canvas 绘制白底黑柱；
- Canvas 只读取已经生成的 256 个归一化高度，不遍历 Bitmap；
- 显示最高灰度频次和 0、128、255 轴标；
- 页面使用可滚动结构，低高度屏幕也能看到直方图和后续结果。

### 3.7 RGB 三通道分析

直方图卡可在“灰度直方图”和“RGB 三通道”间切换：

- 分别统计 R、G、B 的 256 个频次；
- 显示三通道曲线；
- 显示 `avgR / avgG / avgB`；
- 判断主导通道；
- 计算通道差异度；
- 输出色彩均衡、轻微偏红/绿/蓝或明显偏红/绿/蓝；
- 切换只改变展示，不重新解码图片。

RGB 是对灰度亮度分析的补充，不改变灰度直方图课程结果。

### 3.8 图像质量指标

质量指标基于已经生成的 256-bin 灰度频次计算，不重新扫描 Bitmap：

- 平均灰度；
- 暗部占比：灰度 0–63；
- 中间调占比；
- 亮部占比：灰度 192–255；
- 灰度标准差；
- 质量分类：偏暗、偏亮、低对比度或正常。

当前主要规则：

- 平均灰度 `<85` 或暗部占比 `≥60%` 判为偏暗；
- 平均灰度 `>170` 或亮部占比 `≥60%` 判为偏亮；
- 其余图像标准差 `<35` 判为低对比度；
- 其余判为正常；
- 额外检查直方图向黑端或白端明显集中，避免遮挡镜头等极端情况误判正常；
- 判定优先级为偏暗、偏亮、低对比度、正常。

该功能是课程级、规则化辅助判断，不属于专业摄影、医学或工业图像质量诊断。

### 3.9 自然语言综合分析

系统将灰度和 RGB 指标转换为本地中文说明：

- 亮度描述；
- 曝光风险；
- 对比度描述；
- RGB 偏色描述；
- 综合结论；
- 改善建议。

自然语言由固定规则生成，使用“可能”“建议”等表述。整个过程离线运行，不上传图片，也不调用大模型或网络 API。

### 3.10 ROI 局部区域分析

用户可以选择是否分析局部区域：

1. 点击“选择局部分析区域”；
2. 在预览图上拖动矩形框；
3. 点击“确认区域并重新分析”；
4. 系统把预览坐标映射到原始 Bitmap 坐标；
5. 裁剪 ROI，并复用原有灰度、RGB、质量、综合分析和性能链路；
6. 点击“恢复全图分析”可返回原图结果。

坐标映射处理 `ContentScale.Fit`、居中留白、反向拖拽、越界裁剪和过小区域拒绝。ROI 不会覆盖或销毁原图。

### 3.11 分阶段性能展示

性能卡按阶段显示耗时：

- 图片解码；
- Bitmap 像素访问或复制；
- 灰度转换；
- 频次统计或融合统计；
- 0–100 归一化；
- 数组分配和调度开销；
- 多线程结果合并；
- 核心计算总计；
- 执行引擎、工作线程和算法方案；
- 300ms 预算状态。

300ms 的正式口径是“从像素访问开始，到灰度统计和 0–100 归一化完成”。图片解码和 Canvas 绘制单独展示，不混入核心计算结论。

### 3.12 错误恢复与并发保护

- 图片加载、解码和 CPU 计算不在 Compose 重组或 Canvas 中执行；
- 选新图片会取消旧请求，并使用请求编号拒绝过期结果覆盖新图片；
- 取消选择不改变页面；
- 新图片加载失败时保留旧成功状态；
- 加载期间显示进度并避免重复操作；
- 错误页面提供可理解的中文原因和恢复入口；
- Native 失败自动回退 Kotlin；
- 相机权限拒绝、相机绑定失败、保存失败和设备能力不足都有独立状态。

## 四、CameraX 智能拍摄助手

### 4.1 权限与实时预览

- 首页可进入 Camera 页面；
- 首次进入显示相机权限说明和授权按钮；
- 用户拒绝权限时仍能返回首页；
- CameraX Preview 显示实时画面；
- `uses-feature android.hardware.camera` 为 `required=false`，无相机设备仍可安装静态分析功能。

### 4.2 实时亮度直方图

- CameraX ImageAnalysis 使用 `YUV_420_888`；
- 实时阶段直接读取 Y 通道作为亮度信号；
- 使用 `KEEP_ONLY_LATEST`，过期帧不会排队；
- 默认约每 400ms 分析一帧；
- 每帧最终关闭 `ImageProxy`；
- 展示实时亮度直方图、帧尺寸、数据源、最高频次、平均亮度、暗亮占比、标准差和质量分类。

实时 Y 通道用于拍摄前反馈，不作为静态图片课程性能结论，也不冒充课程 RGB 灰度公式结果。

### 4.3 三色画面状态

预览右上角显示：

- 绿色“质量良好”；
- 橙色“可优化”；
- 红色“建议重拍”。

徽章只提供快速决策，详细指标、原因和建议仍在下方卡片展示。

### 4.4 定格分析

用户点击预览画面或“定格分析”后：

- 读取当前 `PreviewView.bitmap`；
- 复制为稳定的 ARGB Bitmap；
- 按课程 RGB 灰度公式重新生成定格直方图；
- 生成质量指标和智能拍摄建议；
- 画面、直方图和建议保持不变，不再被实时帧覆盖；
- 点击“继续实时预览”恢复实时更新。

实时 Y 通道负责流畅反馈，定格 RGB 公式负责稳定结果，两个数据源在界面和文档中明确区分。

### 4.5 保存到系统相册

- 定格画面可通过 MediaStore 保存；
- 保存目录为 `Pictures/ImageHistogramAnalyzer`；
- 不申请额外相册存储权限；
- 保存失败时保留当前定格结果并显示错误；
- 当前保存的是预览定格 JPEG，不是相机传感器原始分辨率照片。

### 4.6 本地智能拍摄建议

规则根据定格直方图生成：

- 严重欠曝、偏暗、轻微偏暗；
- 严重过曝、偏亮、轻微偏亮；
- 低对比度；
- 正常；
- 判断依据；
- 建议动作；
- 曝光补偿变化量；
- 补光灯动作。

规则只输出建议，不会在分析过程中自动修改相机。

### 4.7 按建议重拍

“按建议重拍”遵守真实相机语义：

1. 先定格并分析旧画面；
2. 系统生成曝光或补光灯建议；
3. 只有用户点击按钮后才修改相机参数；
4. 修改完成后重新抓取一张新定格画面；
5. 旧定格图不会被修改或冒充成优化后的图片。

按钮状态：

- 正常画面显示“无需重拍”并禁用；
- 存在设备可执行建议时启用“按建议重拍”；
- 低对比度等无法仅靠曝光/补光灯解决时，提示查看文字建议；
- 设备不支持对应控制时禁用并说明原因。

### 4.8 手动曝光与补光灯

拍摄控制面板默认折叠，避免占用预览和直方图空间。展开后支持：

- 曝光 `+`；
- 曝光 `-`；
- 补光灯开启/关闭；
- 当前曝光索引和设备支持范围；
- 当前补光灯状态；
- 再次定格验证。

曝光范围来自 `CameraInfo.exposureState`，所有调整限制在设备实际范围内。补光灯能力来自 `hasFlashUnit()` 和 `torchState`。系统不会自动覆盖用户手动设置，除非用户再次点击“按建议重拍”。

## 八、下一版测试计划建议结构

本节是制定新测试计划时的建议分类，不代表所有项目已经完成测试。

### 8.1 P0：课程核心正确性

目标：证明任何优化都没有改变课程结果。

建议覆盖：

- 灰度公式与 `.5` 取整边界；
- 黑、白、红、绿、蓝和 Alpha 像素；
- 256-bin 频次和等于像素数；
- 0–100 归一化的零输入、单峰和比例；
- 两种课程方案逐 bin 一致；
- Kotlin v2 与 Native v3 逐 bin 一致；
- ROI 与独立裁剪图片结果一致。

通过标准：所有频次数组完全一致，不接受近似相等。

### 8.2 P0：正式性能验收

目标：以固定 L1 在 Xiaomi 14 上验证两个方案核心计算低于 300ms。

建议统一条件：

- 固定 APK、commit、构建类型和设备；
- 固定 L1 文件 SHA-256；
- 记录版本、引擎、线程数、电量；
- 两种方案各执行一次作为课程主表；
- 如需稳定性附录，额外预热和重复，但不要覆盖主表的一行一 ID 规则；
- 记录解码、各核心分段、核心总计、最高频次和一致性；
- 解码和 Canvas 不计入 300ms，但必须单独保留。

### 8.3 P0：静态图片端到端

建议覆盖：

- 冷启动与首页入口；
- 取消选择；
- JPEG、PNG、WebP；
- 选择后不自动分析；
- 两方案按钮和结果；
- 直方图、质量、RGB、综合分析、性能卡；
- 返回首页后结果保留；
- 连续快速换图，旧结果不能覆盖新图；
- 损坏、无权限、非图片和超限错误恢复。

### 8.4 P1：ROI

建议覆盖：

- 左上、中心、右下区域；
- 正向和反向拖拽；
- 图片留白外拖动；
- 越界裁剪；
- 过小区域；
- 横图、竖图和不同屏幕比例；
- ROI 结果、RGB、质量和性能同步更新；
- 恢复全图后元数据和结果正确；
- 连续切换 ROI 不发生内存明显增长。

### 8.5 P0：CameraX 主流程

建议至少准备以下场景：

| 场景 | 主要检查 |
|---|---|
| 正常室内光 | 绿色或合理状态、无需重拍、实时直方图稳定 |
| 完全遮挡镜头 | 黑端集中、偏暗/严重欠曝、不能显示正常 |
| 强光或灯具直射 | 白端集中、偏亮/过曝提示 |
| 低对比度平面 | 橙色可优化，文字说明光线/构图而非强行曝光优化 |
| 权限拒绝 | 清晰提示，可返回首页，再次进入可授权 |
| 定格后移动镜头 | 定格图、直方图和建议保持不变 |
| 继续预览 | 实时画面和指标恢复更新 |
| 保存相册 | 相册可见，失败时不丢失定格图 |
| 按建议重拍 | 调整参数后生成新图，旧图不被修改 |
| 手动曝光连续调整 | 范围受设备限制，按钮状态正确 |
| 补光灯开关 | UI 与真实 torch 状态一致 |

### 8.6 P1：设备能力和生命周期

建议覆盖：

- 无闪光灯设备；
- 不支持曝光补偿或曝光范围很小的设备；
- 前后台切换；
- 屏幕关闭再打开；
- 快速进入/退出 Camera 页；
- 相机被其他应用占用；
- 定格、保存或调整中返回首页；
- 横竖屏策略按当前产品约束确认；
- Android 23、主流中间版本和 Android 16 的兼容性抽样。

### 8.7 P1：UI、无障碍与易用性

建议覆盖：

- 主要按钮文字与真实动作一致；
- 48dp 最小点击区域；
- 1.0×、1.3×、1.5× 字体；
- 窄屏和 Xiaomi 14 实际屏幕；
- 整页滚动、折叠控制区和屏幕内信息密度；
- 固定 light 在系统 dark mode 下的可读性；
- 图片、直方图和三色状态的语义描述；
- 颜色之外同时提供文字，避免只靠红绿判断。

### 8.8 P1：稳定性与资源

建议覆盖：

- 连续分析 20 张图片；
- 大图与 ROI 反复切换；
- Camera 实时预览 10 分钟；
- 连续定格、继续预览和保存；
- 内存峰值、GC、ANR 和崩溃；
- 后台任务取消和旧结果丢弃；
- Native 失败回退 Kotlin 的可用性。

### 8.9 P2：未来增强回归

如果后续增加 ImageCapture 原图拍照、更多相机参数或新质量指标，需要先补充需求和测试，不应直接改变现有课程口径。每项新功能至少增加：

- 纯逻辑测试；
- UI 状态测试；
- 设备能力分支；
- 原静态分析回归；
- 使用说明与 TODO 同步。

## 十一、已知限制与待确认项

- 最终 17 个图片 ID 的 Xiaomi 14 正式数据表尚未全部填写；
- L1 的最终两方案真机数值不能使用模拟器 CSV 或历史非严格同图截图替代；
- CameraX 保存的是预览定格 JPEG，不是 ImageCapture 原始分辨率照片；
- 实时相机使用 Y 通道，定格和静态图片使用课程 RGB 灰度公式；
- 系统 Photo Picker 的直接拍照入口取决于设备；
- 曝光补偿和补光灯能力取决于具体摄像头；
- 质量判断和拍摄建议是本地规则，不是专业摄影模型；
- 当前不支持变焦、对焦点选择、白平衡、ISO 或快门速度控制；
- 当前不联网、不保存分析历史、不提供账号和云同步。
