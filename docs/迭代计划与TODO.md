### v2.0 真机优化结果（2026-07-07）

本轮使用同一杯子场景，截图中的分辨率、像素数和最高频次均与首轮一致；v2.0 文件名显示为 `1783403529271.jpg`。正式报告仍需固定原始测试文件并记录校验值。

| 阶段 | 优先灰度化 | 统计时灰度化 |
|---|---:|---:|
| 图片解码（核心外） | 113.250ms | 113.250ms |
| Bitmap 批量取像素 | 539.234ms | 499.510ms |
| 灰度转换 | 195.738ms | 与统计融合 |
| 频次统计 | 40.181ms | 173.939ms（融合） |
| 归一化 | 0.251ms | 0.225ms |
| 其他开销 | 0.180ms | 0.060ms |
| 多线程合并 | 未启用 | 未启用 |
| 核心总耗时 | 775.583ms | 673.735ms |

结果判断：

- 相比 v1.0 的 1415.321ms / 1304.234ms，两个方案分别加速约 1.82 倍和 1.94 倍，核心耗时下降 45.2% 和 48.3%，说明 `PERF-11` 有效。
- 两方案最高灰度频次仍为 121,520，截图层面未发现结果漂移；完整正确性继续由 256-bin 与全 RGB 自动化测试保证。
- `Bitmap.getPixels()` 占核心时间约 69.5% / 74.1%，且单独已超过 300ms，是当前首要瓶颈。只并行灰度统计无法让总时间达标。
- 优先灰度化的灰度转换与统计合计约 235.919ms；统计时灰度化的融合阶段约 173.939ms，当前算法计算本身已明显低于像素复制耗时。
- 同图两次 `getPixels()` 相差约 39.7ms，当前仍是单次 Debug/构建类型待确认数据，需要预热和多轮中位数避免偶然波动。
- 12.58MP ARGB_8888 Bitmap、彩色 `IntArray` 各约 48MiB；优先灰度化再增加约 48MiB 灰度 `IntArray`，三者理论驻留约 144MiB，存在 GC 和峰值内存风险。

更新后的优化顺序：先建立可复现多轮基准，再比较像素访问路径并消除 Java 整图复制；之后才并行灰度统计。若 Android/Kotlin 路径仍不能把取像素降到预算内，则使用 NDK `AndroidBitmap_lockPixels` 直接访问 Bitmap，并进一步评估 ARM NEON。所有方案必须包含转换/拷贝成本并通过 256-bin 完全一致性测试。

建议性能预算：像素访问 ≤100ms，灰度转换与统计 ≤150ms，归一化/合并/其他 ≤10ms，核心中位数 ≤260ms，为 300ms 验收线保留约 40ms 波动空间。

### v3.0 本地优化结果（2026-07-07）

本轮不是单点微调，而是替换核心执行引擎：通过 NDK `AndroidBitmap_lockPixels` 直接访问 RGBA_8888 Bitmap，在 C++ 中完成两种课程流程；按行分块、线程私有 256-bin 后归并，避免共享计数竞争。优先灰度化的中间灰度由 4-byte `IntArray` 改为 1-byte native 缓冲；Native 不可用时仍回退 Kotlin v2。界面新增执行引擎、工作线程和 Native 分段耗时。

API 36/arm64 模拟器先全局 JIT 预热 20 次，再按尺寸预热 5 次并正式运行 10 次。12,582,912 像素确定性夹具结果：

| 方案 | Kotlin v2 中位/P90 | Native v3 中位/P90 | 中位加速 |
|---|---:|---:|---:|
| 优先灰度化 | 95.948 / 121.883ms | 30.885 / 32.451ms | 3.11× |
| 统计时灰度化 | 64.455 / 69.353ms | 15.067 / 16.305ms | 4.28× |

该 AVD 的 `nproc=1`，所以 1/2/4/6/8 线程没有稳定缩放收益；这些数字证明的是 Native 直读、内存压缩和 C++ 热循环有效，不用于宣称八核并行收益。真机表、完整模拟器明细和后续版本占位见 `docs/性能优化数据表.md`。

正确性方面，Kotlin 快路径已穷举全部 16,777,216 种 RGB；Native 又覆盖全部 16,782 个精确 `.5` 取整边界、混合 Alpha Bitmap、两种方案的 256-bin 与 0–100 归一化一致性。当前未写显式 ARM NEON，也不把 `-O3` 可能产生的自动向量化宣传为已实现 SIMD。

### v3.0 真机达标结果（2026-07-07）

- 设备：Xiaomi 14，8 个 Native 工作线程。
- 图片：`1783408658743.jpg`，3072×4096，共 12,582,912 像素；两方案最高灰度频次均为 124,455。
- 优先灰度化：解码 117.465ms（核心外），核心计算 56.957ms。
- 统计时灰度化：解码 117.465ms（核心外），核心计算 22.244ms。
- 两方案分别只占 300ms 预算约 19.0% 和 7.4%，单次结果均有充足余量。

决策：v3.0 已满足“核心计算 300ms 内”目标，性能方向进入冻结状态。v2 与 v3 图片的最高频次不同（121,520 与 124,455），因此约 13.62× / 30.29× 的对比仅作为同尺寸、相似场景参考，不冒充严格同文件加速比。正式报告仍在课程验收前用固定原图完成预热和至少 10 次采样。
