# 概要设计与系统架构

## 一、总体架构

项目采用单 `:app` 模块和轻量分层，依赖方向为：

```text
Compose UI
    ↓ state / callbacks
AnalyzerViewModel / CameraViewModel
    ↓
ImageLoader + HistogramCalculator + ImageQualityAnalyzer + CameraX analyzer
    ↓
ContentResolver / Kotlin 算法 / JNI Native v3 / CameraX
```

没有 XML 页面、Navigation Compose、Room、Retrofit、Hilt/Koin、数据库、后端或云服务。

## 三、静态图片主流程

1. `ImageHistogramApp` 启动系统 Photo Picker；
2. 成功 URI 交给 `AnalyzerViewModel.selectImage`；取消则保持当前页面和状态；
3. `ImageRepository` 在 IO 调度器调用 `BitmapDecoder`；
4. 解码成功后 UI 显示预览和元数据，但不自动计算；
5. 用户选择计算策略并点击计算；
6. ViewModel 在 Default 调度器优先调用 Native v3，Native 不可用时回退 Kotlin；
7. 计算返回 `HistogramResult` 和 `HistogramPerformanceMetrics`；
8. `ImageQualityAnalyzer` 只遍历 256-bin 得到质量结果；
9. Compose Canvas 只绘制归一化高度，不读取源图像；
10. UI 展示直方图、RGB、质量、综合分析和 300ms 预算；
11. 如用户进入 ROI 模式，预览框选区域会映射到 Bitmap 坐标并裁剪，再复用同一分析链路生成局部结果；“恢复全图分析”会回到原图 Bitmap。

## 四、实时拍摄主流程

1. 首页进入 Camera 页，用户授权相机；
2. CameraX Preview 负责实时画面，ImageAnalysis 使用 `KEEP_ONLY_LATEST`；
3. 实时链路按约 400ms 节流读取 Y 通道，生成轻量亮度直方图和质量指标；
4. 右上角状态徽章根据最新实时结果显示“质量良好 / 可优化 / 建议重拍”；
5. 用户点击预览或“定格分析”后，读取 `PreviewView.bitmap`，按课程 RGB 灰度公式生成定格直方图、质量、综合建议；
6. 定格画面可保存到系统相册；
7. 拍摄控制面板默认折叠，支持手动曝光补偿和补光灯；
8. “按建议重拍”仅在定格结果存在设备可执行的曝光/补光灯建议时启用，点击后先调整相机，再重新抓取一张定格图。

## 五、状态设计

`AnalyzerUiState` 是页面唯一状态源，包含加载标记、Bitmap、元数据、选中策略、直方图、质量结果、解码时间、性能指标和错误文本。状态向下传递，用户动作通过回调向上传递。

选择新图片时取消旧任务并增加 `requestId`。即使 Native 任务不能即时停止，旧请求结果也不能覆盖新图片。替换图片失败时恢复上一次成功状态，并叠加可恢复错误提示。

`CameraUiState` 是实时相机页唯一状态源，包含权限、绑定状态、实时分析、定格帧、保存状态、相机调节能力、手动控制状态、拍摄建议和错误文本。实时结果与定格结果分开保存，避免用户下滑查看时画面和分析结果继续变化。

## 六、线程与性能设计

- URI/Bitmap 解码：`Dispatchers.IO`；
- Kotlin/Native 核心计算：`Dispatchers.Default`；
- Native：最多 8 个工作线程，按行分块，每线程私有 256-bin，完成后归并；
- UI：主线程只处理状态与绘制；
- 单调纳秒时钟只在阶段边界采样，不进入逐像素循环。
- CameraX 实时分析：保留最新帧、节流分析、关闭 `ImageProxy`，不阻塞预览；
- 定格分析：基于预览 Bitmap 的 RGB 像素运行轻量路径，不冒充静态原图性能验收。

## 七、内存设计

- 解码前验证尺寸，单图上限 1600 万像素；
- ImageDecoder 强制软件 Bitmap，便于稳定读取；
- Native v3 直接锁定 Bitmap，避免约 48MiB Java 彩色 `IntArray`；
- 优先灰度化使用 1 byte/像素的 Native 灰度缓冲；
- UI 只保留当前图片和当前结果，不保存历史图片列表。
- 相机页只保留一张定格预览图和最新实时结果，继续预览时释放定格状态；
- ROI 分析裁剪前校验尺寸，失败时保留原图结果。

## 八、错误处理

| 类型 | 用户行为 |
|---|---|
| 取消选择 | 不跳转、不清空、不显示错误 |
| 无法打开/权限失效 | 提示重新选择，保留旧成功内容 |
| 损坏/格式不支持 | 提示文件可能损坏或格式不支持 |
| 超过 1600 万像素 | 显示实际百万像素和安全上限 |
| Native 不可用 | 自动回退 Kotlin v2，不阻塞主流程 |
| 旧任务完成 | 通过请求编号丢弃过期结果 |
| 相机权限拒绝 | 保持可返回首页，提示授权后开始预览 |
| 曝光/补光灯不支持 | 禁用对应按钮，仍可实时分析和定格 |
| 保存定格图失败 | 显示中文错误，不影响当前定格结果 |

## 九、UI 设计摘要

应用固定使用 B“晴空柑橘”light 方案：蓝色主操作、浅蓝算法/直方图层级、薄荷绿质量模块、淡柑橘性能模块。直方图保持课程要求的白底黑柱。详细 token 和无障碍规则见 [UI Design System](../ui-design-system.md)。

## 十、关键设计决策

| 决策 | 原因 | 结果 |
|---|---|---|
| 保持单 `:app` 模块 | 课程项目规模有限，多模块收益不足 | 构建和交接简单，职责仍由包分层保证 |
| 使用系统 Photo Picker | 减少权限和文件路径兼容问题 | 不申请相册存储权限，Xiaomi 14 可直接拍照 |
| 选图后不自动计算 | 课程要求比较两种明确方案 | 用户先确认图片，再选择算法并主动执行 |
| 质量分析复用直方图 | 避免二次扫描上千万像素 | 只遍历 256 个 bin，且不混入核心耗时 |
| RGB/综合分析作为辅助解释 | 答辩需要说明“直方图如何帮助判断图像” | 在不改变课程灰度核心算法的前提下增强可读性 |
| ROI 复用静态分析链路 | 避免新增第二套算法 | 局部分析与全图分析结果口径一致 |
| 实时相机使用 Y 通道，定格使用 RGB 公式 | 兼顾预览流畅和课程公式一致性 | 实时状态用于辅助取景，定格结果用于用户查看和保存 |
| 三色质量徽章 | 用户不需要每次下滑才能知道是否需要重拍 | 先用状态色提示，再通过质量卡解释原因 |
| Native 主路径 + Kotlin 回退 | 消除 Java 整图复制，同时保留可运行性 | 性能达标且 Native 失败时仍能完成分析 |
| 不引入 Room/Hilt/Retrofit | 没有持久化、网络或复杂依赖图需求 | 避免无关架构和答辩负担 |

## 十一、可维护性约束

- `MainActivity` 只负责主题和 Compose 入口；
- Canvas 只接收归一化数据，不读取 Bitmap；
- 公式、取整、Alpha 和归一化由领域层与测试固定；
- UI 文案、颜色和间距集中使用 Material 3 主题；
- 性能结论必须同时给出输入尺寸、构建、设备、引擎和计时边界。
