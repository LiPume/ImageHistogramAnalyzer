# 算法与性能优化说明

## 一、灰度与直方图定义

每个像素提取 RGB 后计算：

`gray = round(red × 0.299 + green × 0.587 + blue × 0.114)`

结果限制在 `0..255`，并使 `counts[gray] += 1`。频次数组固定 256 项，频次总和必须等于图片像素数。Alpha 不改变 RGB 统计语义。

归一化高度：

`height[i] = round(count[i] × 100 / max(counts))`

最高非空 bin 映射为 100，其他高度限制在 `0..100`。Canvas 使用 256×100 逻辑坐标绘制白底黑柱。

## 二、两种课程方案

### 方案 A：优先灰度化

1. 读取彩色像素；
2. 生成与像素数相同的灰度缓冲；
3. 扫描灰度缓冲统计 256-bin；
4. 归一化并绘制。

特点：流程直观，便于分开观察灰度转换和统计成本；需要中间灰度缓冲。

### 方案 B：统计时灰度化

1. 读取彩色像素；
2. 同一循环中完成 RGB→灰度和 256-bin 累计；
3. 归一化并绘制。

特点：省去中间灰度缓冲和第二次整图扫描，通常更快。

## 三、质量指标

所有指标只扫描 256 个频次：

- 平均灰度：`Σ(gray × count) / N`；
- 暗部占比：`Σcount[0..63] / N`；
- 亮部占比：`Σcount[192..255] / N`；
- 标准差：由加权二阶矩计算；
- 分类阈值见 [需求规格说明](02-需求规格说明.md)。

## 四、性能计时口径

| 指标 | 是否计入 300ms 核心目标 |
|---|---|
| 图片解码 | 否，单独展示 |
| Bitmap 锁定/像素访问 | 是 |
| 灰度转换 | 是 |
| 频次统计/融合阶段 | 是 |
| 归一化 | 是 |
| 多线程合并和其他调度 | 是 |
| Compose Canvas 绘制 | 否，单独作为 UI 帧体验 |

计时使用单调纳秒时钟，仅在阶段边界打点。最终课程验收使用固定原图、固定构建，每个图片 ID 的两个方案各运行一次并记录完整分段；历史模拟器多轮中位数/P90继续作为优化稳定性旁证。

## 五、版本迭代

| 版本/轮次 | 主要改进 | 大图优先灰度化 | 大图统计时灰度化 | 结论 |
|---|---|---:|---:|---|
| v1.0 | MVP Kotlin 基础实现 | 1415.321ms | 1304.234ms | 未达标 |
| v2.0 | `while` 热循环、整数定点快路径 | 775.583ms | 673.735ms | 下降约 45.2%/48.3%，仍未达标 |
| v3.0 | NDK Bitmap 直读、O3 C++、最多 8 线程、线程私有 bins | 56.957ms | 22.244ms | Xiaomi 14 单次达标 |
| v4.0 验收轮次 | 固定 light UI、可视化和错误恢复；仍使用 Native v3 | 38.291ms | 23.763ms | UI 未破坏性能链路 |

以上真机数字是单次证据；v2/v3 不是严格同一原文件，不能把版本比值当作严格同图加速比。最终结论由文档 07 的固定 L1 单行实测填写。

## 六、Native v3 关键优化

1. `AndroidBitmap_lockPixels` 直接访问 Bitmap，取消 Java 整图 `IntArray` 复制；
2. C++20、`-O3` 热循环；不启用会改变浮点语义的 `fast-math`；
3. 按行分块，最多 8 个线程；每线程独立 256-bin，避免共享计数器竞争；
4. 优先灰度化使用 1 byte/像素灰度缓冲，替代 4-byte Java `IntArray`；
5. 统计时灰度化融合转换和计数；
6. Native 不可用时保留 Kotlin v2 回退。

### 瓶颈如何被逐步消除

| 阶段 | 主要瓶颈 | 对应处理 |
|---|---|---|
| v1 | Kotlin 高阶遍历、逐像素转换与大数组操作 | 建立正确性基线和分阶段计时 |
| v2 | 热循环开销下降，但 `Bitmap.getPixels()` 仍复制整图 | `while` 分块、整数定点快路径、取消检查移到分块边界 |
| v3 | Java 彩色 `IntArray` 复制成为主导成本 | `AndroidBitmap_lockPixels` 直接锁定 Bitmap，Native 层分块统计和归并 |

优化的核心不是简单“改用 C++”，而是删除整张彩色像素数组的 Java 侧复制，并把线程私有统计结果压缩到每线程 256 个计数器。两套课程流程仍然分别保留：方案 A 生成 Native 灰度缓冲，方案 B 融合转换与统计。

## 七、正确性证据

- Kotlin 快路径穷举 16,777,216 种 RGB，与原浮点语义一致；
- Native 覆盖 16,782 个精确 `.5` 取整边界；
- 黑白、RGB 主色、透明像素、混合 Bitmap 和两个流程逐 bin 一致；
- 归一化高度、像素总数、最大频次一致；
- 自动化测试结果见 [测试计划与测试用例](06-测试计划与测试用例.md) 和历史 [测试报告](../测试计划与测试报告.md)。

## 八、未采用方案及原因

- 显式 ARM NEON：当前最慢方案仍有较大余量；除非最终 L1 实测重新超过 300ms，否则收益不足以覆盖复杂度；
- OpenCV/libyuv：会增加 Bitmap 转换、APK 体积和 ABI 维护，而自研 Native 已达标；
- 继续优化 Kotlin `getPixels`：已退出 Native 主路径，不再投入低收益调优。

完整模拟器数据见 [性能优化数据表](../性能优化数据表.md) 和 [模拟器性能 CSV](../performance/模拟器性能测试_v3.csv)。

## 九、性能结论边界

- v1/v2/v3 截图中的大图尺寸相同，但最高频次不同，因此版本间倍数只用于解释优化方向；
- 当前最强真机证据是 v3/v4 的 Xiaomi 14 单次结果，均低于 300ms；
- 最终报告只把固定 L1 行写成正式本次验收，缺失数据保留 `[待真机填写]`；
- 解码、核心计算和 Canvas 必须分开陈述，避免用总等待时间掩盖算法指标。
