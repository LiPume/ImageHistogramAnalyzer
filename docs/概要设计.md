# 图像直方图系统概要设计

## 一、当前版本

- 版本范围：v3.0，阶段 3 性能优化。
- 技术栈：Kotlin、Jetpack Compose、Material 3、ViewModel、StateFlow、协程、Android NDK/C++20/JNI。
- Android 配置：minSdk 23、targetSdk 36、compileSdk 36.1。
- 负责人：lzx。

## 二、模块关系

```text
系统 Photo Picker
       ↓ Uri
ImageRepository / BitmapDecoder
       ↓ DecodedImage(Bitmap + ImageMetadata)
AnalyzerViewModel
       ├─ NativeBitmapHistogramCalculator（v3 主路径）
       │      ↓ JNI
       │   AndroidBitmap_lockPixels + C++ 两方案 + 线程私有 bins
       ├─ BitmapPixelReader（Native 不可用时的 v2 回退）
       ├─ PreGrayscaleHistogramCalculator（Kotlin 回退）
       ├─ BaselineHistogramCalculator（Kotlin 回退）
       ├─ HistogramNormalizer
       └─ AnalyzerUiState
              ↓
AnalyzerScreen / stateless components / HistogramCanvas
```

- `data/image`：通过 `ContentResolver` 读取 URI，校验尺寸并解码软件 Bitmap。
- `domain/model`：定义图片元数据和与 UI 解耦的直方图结果。
- `domain/histogram`：定义方案接口，执行两种灰度统计流程和 0..100 归一化。
- `app/src/main/jni`：直接锁定 RGBA_8888 Bitmap；执行普通 C++ 多线程灰度/统计并归并局部直方图。
- `ui/analyzer`：管理异步任务、取消、错误和页面状态。
- `ui/component`：只展示数据；Canvas 不读取 Bitmap、不执行统计。

## 三、核心规则

- 灰度公式：`gray = red * 0.299 + green * 0.587 + blue * 0.114`。
- 取整规则：四舍五入后限制到 `0..255`。
- 单线程优化：通常使用权重 299/587/114 的整数定点快路径；当加权结果精确落在 `.5` 边界时回退原 `Double + roundToInt`，确保与优化前的全部 RGB 输出一致。
- Alpha 规则：Alpha 不参与公式，透明像素按解码后的 RGB 统计。
- 直方图：保存 256 个原始频次，并按最大频次等比例映射到 `0..100`。
- Canvas：使用 256×100 逻辑坐标，屏幕上等比缩放为白底黑柱。
- MVP 图片上限：1600 万像素；超过后显示错误，不静默缩小后冒充原图分析。

两种课程方案：

1. “优先灰度化”：彩色像素 → 完整灰度数组 → 频次统计 → 归一化 → Canvas 绘制。
2. “统计时灰度化”：彩色像素 → 遍历时灰度化并累计频次 → 归一化 → Canvas 绘制。

Kotlin 两方案共用 `GrayscaleConverter` 与 `HistogramNormalizer`；Native 复现相同的定点快路径与精确 `.5` 浮点回退。自动化测试覆盖全部 16,782 个 `.5` 边界颜色、混合 Alpha 和两方案逐 bin 对比。选图只完成解码与预览，只有用户选择方案并点击计算按钮后才执行统计。

## 四、线程与生命周期

- URI 查询和 Bitmap 解码运行在 `Dispatchers.IO`。
- 批量取像素、灰度统计和归一化运行在 `Dispatchers.Default`。
- 每次重新选图会取消上一任务，并通过请求编号防止旧结果覆盖新状态。
- 大图计算按 16384 像素分块，只在块边界检查协程取消；块内使用索引 `while` 热循环，避免逐像素取模、回调和高阶遍历开销。
- v3 主路径最多使用 8 个 native 工作线程，按行分块；每个线程维护独立 256-bin，主线程只在结束时归并，避免原子操作和共享计数器争用。
- `AndroidBitmap_lockPixels` 与 `unlockPixels` 成对调用；JNI 直接返回 256 个频次，不创建整图 Java 彩色像素数组。
- Native 调用结束前不能响应协程的中途取消，但请求编号会阻止旧结果覆盖新图片；真机验证后再决定是否增加可取消原子标志。

## 五、性能口径

- 图片解码：从 URI 解码为 Bitmap，单独计时。
- 核心计算：Kotlin 从 `Bitmap.getPixels()`、Native 从锁定 Bitmap 开始，到 0..100 归一化完成，作为界面 300ms 判断指标。
- 分段计时：使用可注入的单调纳秒时钟，只在阶段边界采样；显示批量取像素、灰度转换/融合统计、频次统计、归一化、多线程合并和未归属的分配/调度开销。
- “优先灰度化”分别记录灰度转换与频次统计；“统计时灰度化”因同一循环完成两个操作，记录为一个融合阶段，不能人为拆成两个虚假数据。
- Canvas/首帧渲染不计入核心计算，后续用独立帧时间工具测量，避免在绘制回调内更新状态干扰结果。
- 性能卡片显示执行引擎与线程数，避免把 Kotlin 复制路径和 Native 直读路径混为同一数据。
- 开发基准用全局 JIT 预热、尺寸预热、多轮中位数、P90 和最差值；最终结论仍使用真机 Release/接近 Release 与固定原图。

## 六、Native v3 内存与编译策略

- 12.58MP Kotlin v2 至少同时存在约 48MiB Bitmap 与约 48MiB Java 彩色 `IntArray`；优先灰度化再增加约 48MiB `IntArray`。
- Native v3 直接读取 Bitmap；统计时灰度化仅保留每线程 256-bin，优先灰度化只增加约 12MiB 的 `uint8_t` 灰度缓冲。
- 固定 NDK `28.2.13676358`，构建 arm64-v8a、armeabi-v7a、x86_64、x86 四 ABI。
- 使用 C++20 和 `-O3`，关闭 `fast-math`，防止浮点重排破坏课程公式边界语义。
- 当前没有手写 ARM NEON；只有真机 Trace 证明 v3 仍由像素算术主导时才进入 SIMD 原型。
